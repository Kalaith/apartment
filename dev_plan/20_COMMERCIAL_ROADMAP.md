# Commercial Release Roadmap ‚Äî Prototype ‚Üí Shippable Game

*Compiled 2026-07-14 from a full-codebase audit (~18,700 LOC across 78 Rust files + 12 JSON content files), the sim-harness balance report, playtest feedback (`feedback.md`), and the dev_plan roadmaps.*

## Where the project actually stands

**Strong foundation.** The core loop (tenants, happiness, economy, upgrades, relationships) is genuinely functional, config-driven, and tested (~55 tests, clean CI with fmt/clippy/test/dual-target builds). The `theme.rs`/`widgets.rs` design system landed, the tenant-dilemma emergent stories work, error handling in production paths is clean (zero unwrap/panic outside test code), and the sim harness gives empirical balance measurement most indies never build.

**The gap to commercial is not "more engine" ‚Äî it's four things:**
1. **Dead wiring** ‚Äî several fully-modeled systems produce zero runtime effect (regulations, reputation, tenant life events, half the narrative effects).
2. **Content volume** ‚Äî 5 archetypes / 3 buildings / 7 relationship events / 5 missions / 2 dialogues repeat within a single 36-month run.
3. **Presentation** ‚Äî zero audio, minimal juice, no tooltips/settings/confirmations, missing-glyph emoji, dead i18n scaffolding.
4. **Release engineering** ‚Äî no save versioning, achievements broken in shipped builds, no LICENSE/icon/installer/store integration.

## 2. Wire or cut: systems that exist but do nothing

This is the highest-leverage design work in the project ‚Äî features that "look done in the type system" but are inert at runtime. For each: either wire it into gameplay or delete it (per the no-dead-code standard). Wiring is recommended for the first four; they're exactly the "teeth" the balance needs.

- [x] **Regulations / ComplianceSystem is entirely dead.** ~~no code path ever schedules an inspection or creates a violation.~~ **WIRED (2026-07-14).** Added a data-driven `RegulationsConfig` (`assets/config.json` ‚Üí `regulations` block: pass threshold, random-inspection chance, fine multiplier, fix-deadline months, reputation deltas). New `ComplianceSystem::run_inspection` grades each regulation against the building's inspection score (min of average unit condition and hallway condition): below the threshold ‚Üí citation, fine, `pending_fixes` deadline, and compliance-reputation loss; clean ‚Üí reputation gain. `RegulationType::inspection_interval`/`name` extracted; `add_violation` de-test-gated. Orchestrated each month in `gameplay_turn::run_due_inspections` ‚Äî runs scheduled (timer-driven) and random spot-check inspections on the active building, bills accrued fines (including missed-deadline escalations from `ComplianceSystem::tick`) as a required `InspectionFine` expense (can push toward bankruptcy ‚Äî real teeth), and moves the neighborhood's visible reputation. 5 new tests. This directly attacks "too easy to make money": neglect now carries scheduled, escalating cost.
- [x] **Half the narrative effect types are silently dropped.** ~~`state/gameplay_effects.rs:63` `_ => {}` catch-all discards 6 of 12 variants.~~ **WIRED (2026-07-14).** Implemented all six handlers and **removed the catch-all** so any future variant is now a compile error, not a silent drop: `NeighborhoodReputation`/`RentDemand` mutate the matching neighborhood (clamped), `EconomyChange` moves `city.economy_health` (0.5‚Äì1.5), `BuildingHappiness` shifts the active building's tenants, `PropertyValue` scales the building's rent ceiling (`rent_multiplier`) as the value proxy, and `TriggerInspection` drives a complaint inspection through the item-2.1 engine (`execute_inspection` + immediate `bill_outstanding_fines`, both refactored out of the monthly path for reuse). City/neighborhood news events now have real consequences instead of being pure flavor. 5 new tests.
- [x] **Reputation is displayed and scored but cannot be moved by the player.** ~~`choice.reputation_change` never applied; `MissionReward::Reputation` shows floating text only.~~ **WIRED (2026-07-14).** Reputation is now a real, player-movable currency with sources, a write path, and a consequence: **Sources** ‚Äî `NarrativeEventSystem::process_choice` now returns a `ChoiceOutcome { effect, reputation_change, neighborhood_id }`, and `gameplay_actions` applies the reputation change on every event resolution; `MissionReward::Reputation` now actually moves reputation (was floating-text only); and failed/clean inspections move it (item 2.1). **Write path** ‚Äî new `apply_reputation_change(delta, neighborhood_id)` mutates the targeted (or active) neighborhood, clamped 0‚Äì100, with floating-text feedback. **Consequence** ‚Äî applicant *volume* now scales with the active neighborhood's reputation via a data-driven `applications.reputation_influence` (neutral 50 ‚Üí √ó1.0, 0 ‚Üí √ó0.5, 100 ‚Üí √ó1.5), threaded through `advance_tick` ‚Üí `generate_applications`. Reputation now feeds a live gameplay loop (cultivate it ‚Üí more applicants ‚Üí better tenant selection) instead of being a static scored stat. 4 new tests.
- [x] **Tenant life events (the emergent-story engine) have no generator.** ~~10 `LifeChangeType`s modeled but nothing generates them and `StoryImpact::LifeChange` is a no-op.~~ **WIRED (2026-07-14).** `LifeChangeType::impact()` maps each of the 10 life changes to concrete consequences (happiness / rent-tolerance / move-out-risk) composed from a **data-driven `LifeEventsConfig`** (`assets/config.json` ‚Üí `life_events`: monthly chance + reusable magnitudes), and `LifeChangeType::eligible_for(archetype)` gates which are plausible per archetype. `apply_story_impact` now **expands the former `LifeChange` no-op** into those effects. New `state/gameplay_life_events.rs::generate_tenant_life_events` runs each month in `end_turn`: rolls the per-tenant chance, applies the impact, records a `StoryEvent` on the tenant's story, and surfaces the emergent beat via event log + floating text ("Sarah lost their job."). This is the engine behind the playtest's most-praised hook ‚Äî job losses, new babies, and retirements now create the income/space pressure that forces real keep-or-evict dilemmas. 4 new tests.
- [x] **Missions: 3 of 6 goal types are never evaluated; `UnlockBuilding` reward is a stub.** ~~`MaintainHappiness`/`PerfectCollection`/`FullRepair` fell to `_ => {}`; `MissionReward::UnlockBuilding` did nothing.~~ **WIRED (2026-07-14).** `update_missions` now evaluates all six goals and the catch-all is removed: `MaintainHappiness` accrues consecutive months at/above the happiness threshold (resets on a bad month), `PerfectCollection` accrues months with no missed-rent event (reads `last_tick_result`, which `end_turn` now records *before* mission evaluation), and `FullRepair` completes when every unit and the hallway are at/above the repair bar. `MissionReward::UnlockBuilding(order)` now unlocks the matching building template in persistent player progress via new `unlock_building_by_order`. 2 new tests. **Deferred (UI, not dead logic):** missions still auto-accept ‚Äî turning acceptance into a player choice needs a mission-select panel, tracked for the ¬ß6 UI milestone (making it manual now, with no UI, would just prevent missions from ever starting).
- [x] **Dialogue system: 1 of 3 types live.** ~~Only `FaceToFaceRequest` generated; `ConflictMediation`/`RentNegotiation` dead.~~ **WIRED (2026-07-14).** `generate_dialogues` now also produces both: `ConflictMediation` is sourced from a real **hostile relationship** in the tenant network (initiator complains about a housed neighbor; choices mediate / take-a-side / stay-out via the existing `RelationshipChange`/`HappinessChange` effects), and `RentNegotiation` fires when the building charges above baseline (`rent_multiplier > 1.1`) and a price-sensitive tenant (Elderly/Family/Student) is unhappy ‚Äî offering a one-time rent break or holding firm. The network is threaded into the generator; all five `DialogueEffect` variants were already handled by the dispatcher. 1 new test.
- [ ] **Ownership/HOA voting is placeholder** ‚Äî "satisfaction > 50 = YES" (`building/ownership.rs:81,94`) and the UI literally renders "Management options not yet implemented for this ownership type" (`ui/ownership_panel.rs:285`). Either finish condo/HOA as a real late-game mechanic or cut the surface area for 1.0.
- [x] **Tenant council has no mechanical effect** ‚Äî ~~`should_form_council` fires floating text only.~~ **WIRED (2026-07-14).** A newly-formed council now takes a one-time collective action: it bargains the building's `rent_multiplier` down (data-driven `gentrification.council_rent_rollback`) and solidarity lifts tenant morale (`council_solidarity_happiness`). Latched via a new `council_formed` state flag so the effect applies once and can re-form if the landlord backslides. Pushing rent too hard on an unhappy building now has a concrete cost.
- [x] **Developer counter-offer is a stub** ‚Äî ~~`effect: None`.~~ **WIRED (2026-07-14).** "Counter Offer" is now a gamble decided at event generation: ~60% the developer sweetens the deal ~25% (Money + SellBuilding), ~40% they walk and the sale is off. No more inert button.
- [x] **Dead animation:** ~~`panel_tween.current` never read.~~ **WIRED (2026-07-14).** The detail panel (apartment/hallway) now slides in from the right as the selection tween eases to 1.0 (`offset_x = (1 - panel_tween.current) * 60`).
- [~] **Dead config to reconcile:** (partial ‚Äî 2026-07-14)
  - [x] `happiness.design_style_modifiers` now covers all five tiers ‚Äî **Luxury (+15) and Opulent (+20) added** (config.json + `config_defaults.rs`); those upgrades finally grant the happiness they should.
  - [ ] `text_strings.json` never loaded / out of sync ‚Äî deferred to ¬ß8 (localization/string-table work).
  - [ ] `ThemeConfig`/`LayoutConfig` loaded but never read ‚Äî deferred (thread `Theme::from_config`, per the UI notes).
  - [ ] `config.json` `economy.design_upgrade_costs` / `soundproofing_cost` / `kitchen_renovation_cost` are stale duplicates of `upgrades.json` ‚Äî left in place per "wire, don't delete" instruction; revisit when the config surface is cleaned up.
  - Two divergent final-score formulas exist (`win_condition.rs:64` vs `career_summary.rs:31`) ‚Äî unify.

## 3. Game design & balance (the "too easy" problem)

**¬ß3 COMPLETE (2026-07-14).** All seven items done. The strategy ranking **inverted** over the pass: at the start the harness reported Greedy $103k > Investor $90k > Slumlord $88k with **0 bankruptcies across 180 runs**; it now reports **Investor $96k ‚â• Greedy $95k ‚â´ Slumlord $20k (25/60 bankrupt, negative steady net)**. Diligent management now wins, adequate play coasts less, and neglect is genuinely loseable ‚Äî the "too easy" problem is substantially addressed, with every knob data-driven in `config.json` and re-verifiable via `cargo test balance_report -- --ignored --nocapture`. (Original starting state, for reference: Greedy ~20√ó starting funds, 0 upgrades needed; playtest: no reason to reject tenants, staff impact invisible, no reason to sell.)

- [x] **Make failure reachable.** ~~Zero bankruptcies in 180 runs; even never-repairing Slumlord finished comfortably.~~ **DONE (2026-07-14).** Two parts: (1) the ¬ß2 regulatory teeth (condition-gated inspection fines) lived in `end_turn`, so the harness ‚Äî which drives `advance_tick` directly ‚Äî couldn't see them; added `run_inspections_and_fines` to the harness so it now measures them (deterministic per seed). (2) Bumped `regulations.fine_multiplier` 1.0‚Üí1.25. **Measured result (60 seeds):** never-repairing **Slumlord goes from 0/60 ‚Üí 9/60 bankruptcies**, finishes at **$32,961 vs Greedy's $103,704**, steady net collapsing from ~$2,239/mo to ~$262/mo ‚Äî while reasonable play (Greedy/Investor, condition kept above the citation threshold) is **unaffected at 0/60**. Failure now punishes neglect, not attention. Verified via `cargo test balance_report -- --ignored --nocapture`. *(Greedy's low-effort dominance ‚Äî the "too easy for adequate play" ceiling ‚Äî is addressed by the reward-side items below, not the failure floor.)*
- [x] **Make tenant selection matter.** ~~Vetting works but nothing forces trade-offs; no reason to reject anyone.~~ **DONE (2026-07-14).** Built the reward/risk structure that makes screening a real decision, all data-driven in `tenant_risk`: (1) **temptation** ‚Äî risky applicants (low reliability/behavior) now carry a rent-tolerance premium (`risky_rent_premium_percent`, scales with how risky they are), so they accept readily and pay more ‚Äî a bad tenant is *tempting*; (2) **a pool worth screening** ‚Äî a data-driven minority of applicants are forced into the problem-tenant range regardless of archetype (`problem_applicant_chance_percent: 18`), since the archetype base reliabilities were too high for screening to catch anyone; (3) **cost** ‚Äî skip-rent nudged up (`skip_rent_chance_percent` 20‚Üí25), and the item-3.1 inspection fines compound tenant damage into citations. **Measured (60 seeds):** accept-all Greedy fell $103k‚Üí$97.5k with lower steady net (accepting everyone now has a real price), the vetting strategy is nearly tied despite its heavy upgrade spend, and neglect-plus-bad-tenants (Slumlord) is catastrophic ‚Äî steady net ~$102/mo, 16/60 bankrupt. Rejecting also has a vacancy cost via the reputation-gated applicant volume (item 2.3). 2 new tests.
- [x] **Make staff visibly worth it.** ~~Mechanics exist but the playtest found them unnoticeable.~~ **DONE (2026-07-14).** Two parts: (1) **legibility** ‚Äî `process_janitor_maintenance` now tallies the exact repair cost its upkeep spared (condition restored √ó per-point cost) and emits a monthly event ("Janitor upkeep offset ~$165 of wear this month"), so the value the player was paying for is finally on screen; (2) **pricing** ‚Äî the janitor cost $200/mo but only saved ~$165 in direct repairs (net-negative on visible math, hence "worthless"), so salaries were cut to a defensible level (`staff_costs`: janitor 200‚Üí150, security 400‚Üí320, manager 600‚Üí480). **Measured (60 seeds):** the staff-hiring Investor's steady net now *exceeds* accept-all Greedy's ($3,039 vs $2,922) and its final cash nearly caught up ($97.7k vs $99.1k, gap ~$8k‚Üí~$1.4k) ‚Äî diligent management now pays. Neglect (Slumlord, hires no staff) is unaffected by the cut and stays punished (20/60 bankrupt).
- [x] **Give selling a purpose.** ~~"No reason to sell" ‚Äî cash was useless (money too easy) and selling cost occupancy/score.~~ **DONE (2026-07-14).** Two forces now give the `SellUnitAsCondo` lever real purpose: (1) **defensive liquidity** ‚Äî the ¬ß3.1‚Äì3.3 teeth make cash crunches real (inspection fines + bad tenants can push toward bankruptcy), so selling a unit is a genuine emergency lifeline; (2) **market timing** ‚Äî the condo sale price is now market-responsive (`condo_sale_market_multiplier` = `economy_health √ó (1 + gentrification √ó condo_sale_boom_bonus)`, data-driven, clamped 0.4‚Äì2.5√ó), so cashing out a unit during an economic boom in a gentrifying neighborhood is a lucrative, deliberate play instead of a flat payout. The live price is surfaced on the Sell button (ownership panel takes the multiplier) so the player can *see* when the market is hot. 1 new test. *(Full portfolio/re-acquisition play remains a ¬ß4 item; this makes the existing sell mechanic a real decision within the current single-building game.)*
- [x] **Difficulty settings.** ~~3 tiers differ only in unit count/condition/rent.~~ **DONE (2026-07-14).** Added a data-driven `difficulty` config block (`DifficultyModifiers` keyed by the template's "Easy"/"Medium"/"Hard") applied once in `new_with_template` via `GameConfig::apply_difficulty`, turning the tiers into genuinely different games: **starting funds** (Easy 7000 / Medium 5000 / Hard 3500 ‚Äî and this finally makes starting cash tier-dependent instead of a hardcoded 5000), **regulation strictness** (fine multiplier √ó0.75/√ó1.0/√ó1.5, random-inspection chance 5/8/12%), **problem-tenant rate** (10/18/28%), and **overhead** (base monthly cost per unit √ó0.85/√ó1.0/√ó1.15). Unknown tiers no-op to a safe 5000/baseline. 3 new tests. **Deferred (UI):** a difficulty selector *independent* of building choice ‚Äî the campaign is intentionally difficulty-progressive (Easy‚ÜíMedium‚ÜíHard buildings), so an orthogonal selector is a menu-UX addition for the ¬ß6/¬ß8 UI pass, not dead logic.
- [x] **Per-run seeded RNG.** ~~No seed stored; RNG never seeded.~~ **DONE (2026-07-14).** The game **never called `srand`**, so the shared RNG ran from a fixed default ‚Äî every playthrough was actually *identical*. Now `GameplayState` stores a `seed` (serialized); `new_with_template` picks one from wall-clock entropy (via `macroquad::miniquad::date::now`, which works on native and wasm ‚Äî `std::time` panics on wasm) and seeds the RNG before any generation, so **runs now genuinely vary**. New `new_with_template_seed(config, template, seed)` starts a reproducible run from an explicit seed (daily-challenge / bug-report / test basis). `post_load` re-seeds from the saved seed, so **reloading a save can't reroll** future outcomes (a clean save-scumming policy), and ‚Äî a bug this surfaced ‚Äî also re-applies the item-3.5 difficulty modifiers, which were being silently dropped on load because `config` isn't serialized. 1 determinism test. *(Full bit-exact save/load replay would need a state-owned `SeededRng` instead of the global one ‚Äî a larger refactor noted for later; the toolkit already ships a serializable `SeededRng` for it.)*
- [x] **Pacing curve.** ~~Full occupancy by month ~6, then 30 months of autopilot.~~ **DONE (2026-07-14).** Made critical failures a data-driven **aging model** (`critical_failures` config): both failure probability and repair cost climb each year the building ages (year 1 ‚Üí 3: prob 5‚Üí15 per 1000 per type, cost +$0‚Üí+$700), and the two hardcoded failure types (boiler $1500 / structural $2500) are now config-driven. The late game stops coasting ‚Äî expenses become variable and rising, so the player must keep reserves and consider security (which halves failure odds). **Measured (60 seeds):** Greedy's steady net eroded ($2,922‚Üí$2,813) and ‚Äî the payoff ‚Äî the security-hiring **Investor now overtakes accept-all Greedy** ($96,057 vs $95,474), while neglect (Slumlord) hits a *negative* steady net and 25/60 bankruptcies. 1 new test.

## 9. Save system & persistence robustness

Native saves are already commercial-grade in *location and atomicity* (`%LOCALAPPDATA%\apartment_manager\`, temp-file+rename). The gap is durability across updates ‚Äî the #1 post-launch risk.

- [ ] **Save versioning + migration.** `GameplayState` has no version field; the toolkit's migration API (`load_from_slot_with_migration`, `peek_version_value`) exists and is unused; only 3 of ~20 core state fields have `#[serde(default)]`. **Any added non-optional field in any core struct breaks every existing save.** Add a version field now (before launch), use the toolkit migration path, and default-annotate aggressively.
- [ ] **Fail loudly, not silently.** A failed load currently `eprintln!`s and leaves the Continue button rendered-but-dead (`state/menu.rs:67-71`) ‚Äî silent data loss from the player's perspective. Show an error, keep a backup copy of the last-good save, offer "start fresh."
- [ ] **Multiple save slots** ‚Äî toolkit `save_to_slot`/`get_save_slots` API exists unused; one slot means one run per player per machine.
- [ ] **Save-system tests** ‚Äî currently 1 round-trip test. Add: old-schema fixture loads (freeze a pre-release save as a fixture), corrupted-file handling, version-migration paths.
- [ ] **Steam Cloud** (if Steam, ¬ß10): the AppData location makes this a config-only step, but plan for it.
